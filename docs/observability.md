# Observability

FrigateRelay emits OpenTelemetry metrics and traces via the `Meter "FrigateRelay"` and
`ActivitySource "FrigateRelay"` primitives, plus structured Serilog logs to console and
optionally to a Seq instance. This document covers the full counter inventory, cardinality
rules, how to enable OTLP export, and how to run the reference observability stacks.

---

## What is emitted

FrigateRelay publishes three observable signal types:

| Signal | Protocol | Instrument |
|---|---|---|
| Metrics (counters) | OTLP/gRPC | `Meter "FrigateRelay"` — 10 counters listed below |
| Distributed traces | OTLP/gRPC | `ActivitySource "FrigateRelay"` — spans propagated across the channel hop |
| Structured logs | Serilog console sink + optional Serilog HTTP sink (Seq) | `ILogger<T>` throughout |

All metrics are `Counter<long>` instruments and follow the `frigaterelay.*` naming prefix.
The .NET runtime metrics (`System.Runtime` meter) are also enabled via
`OpenTelemetry.Instrumentation.Runtime` automatically — they are separate from the
FrigateRelay-specific counters documented here.

Distributed traces: spans are started in `EventPump` and propagated through the
`DispatchItem.ParentContext` field into the action-dispatch pipeline. To receive traces,
configure the same `Otel:OtlpEndpoint` as for metrics (the same OTLP endpoint receives both).

## Counter inventory

The 10 counters below are the complete set emitted by `DispatcherDiagnostics`. The PLAN-2.3
drift test parses this table to verify it matches the counters declared in source — do not
reformat or add columns without updating the test.

| Metric | Tags | Description |
|---|---|---|
| `frigaterelay.events.received` | `camera`, `label` | MQTT event accepted from the source. |
| `frigaterelay.events.matched` | `subscription`, `camera`, `label` | Event matched a configured subscription. |
| `frigaterelay.actions.dispatched` | `subscription`, `camera`, `action` | Action enqueued for execution. |
| `frigaterelay.actions.succeeded` | `subscription`, `camera`, `action` | Action completed without throwing. |
| `frigaterelay.actions.failed` | `subscription`, `camera`, `action` | Action threw after Polly retries exhausted. |
| `frigaterelay.validators.passed` | `subscription`, `camera`, `validator`, `action` | Validator returned `Verdict.Allow`. |
| `frigaterelay.validators.rejected` | `subscription`, `camera`, `validator`, `action` | Validator returned `Verdict.Reject` — short-circuited the action. |
| `frigaterelay.dispatch.drops` | `subscription`, `camera`, `reason` | Bounded channel evicted oldest pending dispatch. |
| `frigaterelay.dispatch.exhausted` | `subscription`, `camera`, `action` | Polly retries exhausted before action succeeded. |
| `frigaterelay.errors.unhandled` | `component` | Unhandled exception in a host component. |

In Prometheus, dots are converted to underscores and `_total` is appended. For example,
`frigaterelay.events.received` becomes `frigaterelay_events_received_total`.

## Cardinality rules

Cardinality is the number of unique tag-value combinations a counter can accumulate. Each
combination consumes memory in the OTel SDK, Prometheus, and Grafana. Unbounded cardinality
causes resource exhaustion.

**Rule 1 — `event_id` is forbidden as a counter tag.**
`EventContext.EventId` is a UUID generated by Frigate for each detection event. Using it as a
tag value creates one time series per event — effectively unbounded. Never pass `event_id` to a
counter tag. The CI gate `git grep '"event_id"' src/FrigateRelay.Host/Dispatch/` must return
empty.

**Rule 2 — Allowed tags are bounded by operator config.**
The tag values used in the counters above are:
- `camera` — bounded by the set of cameras in Frigate config.
- `label` — bounded by the set of object classes tracked in Frigate config.
- `subscription` — bounded by the number of subscriptions in `appsettings.json`.
- `action` — bounded by the set of registered action plugins.
- `validator` — bounded by the set of registered validation plugins.
- `reason` — a fixed enum-like string set (currently only `"channel_full"`).
- `component` — a fixed set of pipeline component names (`"EventPump"`, `"ChannelActionDispatcher"`).

**Rule 3 — Plugin authors follow the same rules.**
If you add counters in a plugin that uses the `Meter "FrigateRelay"` (or a plugin-specific
meter), use the same bounded-tag discipline. Do not use request IDs, session tokens, or any
value whose cardinality grows with event volume as a tag. Document your tag set in your
plugin's README. Prefer tag values sourced from `EventContext.Camera`, `EventContext.Label`,
or plugin-specific config properties.

## How to enable OTLP export

By default, FrigateRelay does not export telemetry — it only emits metrics internally.
To activate OTLP export, set the collector endpoint:

**Option A — appsettings.json or appsettings.Local.json:**

```json
{
  "Otel": {
    "OtlpEndpoint": "http://otel-collector:4317"
  }
}
```

**Option B — environment variable:**

```env
Otel__OtlpEndpoint=http://otel-collector:4317
```

The double-underscore convention maps to the nested `Otel:OtlpEndpoint` key per standard
.NET `IConfiguration` binding rules. The casing matches the JSON section/key on purpose —
copy-paste between the two forms is mechanical.

The OTel SDK also recognises the native `OTEL_EXPORTER_OTLP_ENDPOINT` env var directly. In
FrigateRelay, `HostBootstrap.cs` configures the exporter from `IConfiguration`, so
`Otel__OtlpEndpoint` (or its JSON-form equivalent) takes precedence. Use the .NET form for
FrigateRelay-managed config; reach for the SDK form only if you have a sidecar-injected
OTel config that bypasses `IConfiguration` entirely.

**Protocol:** The exporter defaults to OTLP/gRPC. Port `4317` is the standard gRPC receiver
port. If your collector only exposes the HTTP/protobuf endpoint, set the protocol explicitly:

```env
OTEL_EXPORTER_OTLP_PROTOCOL=http/protobuf
Otel__OtlpEndpoint=http://otel-collector:4318/v1/metrics
```

**What is exported:** When `OtlpEndpoint` is set, both metrics and traces are exported to the
same endpoint. Logs are not routed through OTLP — they go through Serilog sinks (see Seq
section below).

## Reference observability stacks

Two compose stacks ship in `docker/observability/` (landing in this PR via PLAN-2.2).

### Metrics + traces stack (OTel Collector → Prometheus → Grafana)

File: `docker/observability/docker-compose.yml`

This stack wires FrigateRelay → OTel Collector → Prometheus scrape → Grafana. Start it
alongside your FrigateRelay container:

```bash
docker compose -f docker/observability/docker-compose.yml up -d
```

Then set `Otel__OtlpEndpoint=http://otel-collector:4317` in your FrigateRelay environment
(or in the `Otel:OtlpEndpoint` config key). The OTel Collector exposes the standard gRPC
receiver on port `4317` and forwards scraped metrics to Prometheus. Grafana is available at
`http://localhost:3000` with no authentication in the reference stack — restrict access
before exposing outside a local network.

To import the starter dashboard, see [Grafana dashboard](#grafana-dashboard) below.

To run a pre-release smoke validation of the full stack, use:

```bash
make verify-observability
```

The `Makefile` target starts the OTel Collector / Prometheus / Grafana stack, waits for
Prometheus and Grafana to report ready (`/-/ready` and `/api/health` respectively), then
tears the stack down. The recipe verifies the *backend* stack is healthy — it does not
launch a FrigateRelay process. To confirm a tagged counter sample reaches Prometheus end
to end, run a FrigateRelay instance pointed at the collector before invoking the target
and watch for any `frigaterelay_*_total` series at `http://localhost:9090/graph` while
the stack is up. See [RELEASING.md](../RELEASING.md) for when to run this in the release
checklist.

### Logs stack (Seq)

File: `docker/observability/docker-compose.seq.yml`

FrigateRelay can forward structured logs to a Seq instance via the Serilog HTTP sink. Start
the Seq compose stack:

```bash
docker compose -f docker/observability/docker-compose.seq.yml up -d
```

Then set the Seq server URL in your FrigateRelay config:

**Option A — appsettings.json or appsettings.Local.json:**

```json
{
  "Serilog": {
    "Seq": {
      "ServerUrl": "http://localhost:5341"
    }
  }
}
```

**Option B — environment variable:**

```env
SERILOG__SEQ__SERVERURL=http://localhost:5341
```

Seq is available at `http://localhost:5341` (or `http://localhost:8081` for the alternate-port UI mapping) in the
reference stack. The Seq stack is independent of the OTel stack — you can run one, the other,
or both.

**Docker networking note:** The default `http://localhost:5341` works when FrigateRelay
runs natively on the host (or in any container with `network_mode: host`). If FrigateRelay
runs as a container on the same Compose network as the `seq` service, swap `localhost` for
the service name `seq` (e.g., `http://seq:5341`).

## Grafana dashboard

A starter dashboard is provided at `docs/grafana/frigaterelay-dashboard.json`. It covers the
most actionable metrics from the counter inventory.

### Import steps

1. Open Grafana (default: `http://localhost:3000`).
2. Click the **+** icon in the left sidebar and select **Import dashboard**.
3. Click **Upload JSON file** and select `docs/grafana/frigaterelay-dashboard.json`.
4. On the import screen, set the **Prometheus** datasource to your Prometheus instance.
5. Click **Import**.

The dashboard uses a `${DS_PROMETHEUS}` template variable so it adapts to any Prometheus
datasource name on import.

### Panels included

- **Events received** — rate of `frigaterelay_events_received_total` by `camera` and `label`.
- **Events matched** — rate of `frigaterelay_events_matched_total` by `subscription`, `camera`, and `label`.
- **Actions succeeded vs failed** — rate of `frigaterelay_actions_succeeded_total` and
  `frigaterelay_actions_failed_total` paired by `camera` and `action`.
- **Actions dispatched** — rate of `frigaterelay_actions_dispatched_total` by `subscription`
  and `action`.
- **Validator rejections** — rate of `frigaterelay_validators_rejected_total` by `validator`
  and `action`.
- **Dispatch drops** — rate of `frigaterelay_dispatch_drops_total` by `subscription`,
  `camera`, and `reason` — alert-candidate panel.
- **Dispatch exhausted** — rate of `frigaterelay_dispatch_exhausted_total` — alert-candidate
  panel.
- **Unhandled errors** — rate of `frigaterelay_errors_unhandled_total` by `component` —
  alert-candidate panel.

## `make verify-observability`

The `Makefile` target `verify-observability` runs a pre-release smoke check of the
OTel/Prometheus/Grafana stack. It starts the compose stack, waits for Prometheus and
Grafana to report healthy (`/-/ready` and `/api/health` respectively), then tears down.
The recipe verifies the *backend* stack — to confirm a tagged counter sample reaches
Prometheus end to end, run a FrigateRelay instance against the collector beforehand and
watch `:9090/graph` for any `frigaterelay_*_total` series. Run once before tagging a
release:

```bash
make verify-observability
```

See [RELEASING.md](../RELEASING.md) for the full pre-release checklist.
