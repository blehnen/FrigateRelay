# Shipyard Lessons Learned

## [2026-05-08] Phase 16: v1.3.0 Minor Release (#18 + #22 + #30)

### What Went Well

- **CONTEXT-16's pre-research design decisions (D1–D9) prevented mid-flight scope changes.** PLAN dispatch had concrete answers to every architectural question (caller-side normalization vs `Counter<T>.Add` wrapping, `OrdinalIgnoreCase` matching, cameras-only scope, atomic 3-file commit, etc.). Builder agents asked zero design clarification questions — they only got stuck on tool budget, never on intent.
- **CRITIQUE step's RC-4 baseline-compile gate proved load-bearing.** The critic flagged that `CounterIncrementTests.cs:418-421` `EnqueueAsync` call should be verified to compile at HEAD before PLAN-1.1 dispatch. The check was easy (it did compile), but the discipline of running it caught an unrelated risk pattern that future phases will hit.
- **Atomic 3-file commit (D6) for `#30` was clean and verifiable.** `git log --oneline -1 -- <three paths>` showing a single hash is the right invariant — no special tooling needed, no staged-changes orchestration, just a careful `git add` + `git commit`.
- **Sequential dispatch with shared-file ordering kept conflicts manageable.** PLAN-1.1 + PLAN-1.2 both touched `CounterIncrementTests.cs` and `EventPumpSpanTests.cs`; sequential dispatch with PLAN-1.1 first (constructor injection) then PLAN-1.2 (polling-site replacements in the same methods) eliminated merge friction completely.

### Surprises / Discoveries

- **`validators.rejected` is a terminal state when validators short-circuit an action.** The PLAN-1.2 MeterListener fallback for `RunDispatcherAsync` initially used only `actions.{succeeded,failed,exhausted}` as terminal-metric names. The validator-rejected test (`Verdict.Fail`) timed out because `validators.rejected` is the only counter that fires on the validator-short-circuit path — no `actions.*` counter ever increments. The pre-commit test sweep caught the bug before commit.
- **Deterministic-wait switch yielded ~2× test speedup** (Host.Tests 13s → 7.5s after the four `Task.Delay` polling sites were replaced). Fixed-time `Task.Delay` always pays the worst case; deterministic waits short-circuit on the first matching log entry. Real CI-time savings.
- **The `MeterListener`-based `TaskCompletionSource` fallback is the right idiom for log-less terminal signals.** When a code path emits no log message (e.g. dispatcher success), logger-polling can't deterministically wait. Attaching a `MeterListener` to the terminal counter set + a `TaskCompletionSource` that resolves on the first measurement is the cleanest pattern. Authorized inline at exactly one site (`RunDispatcherAsync`); other sites use `WaitForEntriesAsync`.
- **Builder tool-budget caps recur on TDD-heavy + multi-fixture tasks.** Phase 9 saw this pattern (OTel work) and Phase 16 reproduced it: every plan that touched both production code and 4+ test fixtures hit a budget cap mid-task. Mitigation that worked: orchestrator runs the build for the agent on sandbox-blocked tools, feeds errors back via SendMessage, lands the last 1-2 sites inline if needed.

### Pitfalls to Avoid

- **Inserting required constructor parameters before existing optional/nullable trailing parameters breaks any positional caller.** `ChannelActionDispatcher` gained `MetricsTagWriter` at position 4, pushing `ISnapshotResolver?` to position 5. One positional test caller broke with `CS1503`. Two safe approaches: (a) update positional callers to use named arguments (`snapshotResolver: null`), or (b) insert the new arg in the correct slot. Captured in CLAUDE.md Conventions for Phase 17+.
- **MeterListener terminal-state sets must enumerate every code branch, not just the happy path.** Initial fallback covered success/failure/exhaustion but not the validator-short-circuit terminal state. The general lesson: when designing a "wait until done" condition, ask "if every code branch terminates, would my wait condition fire?" before declaring it deterministic.
- **The `git grep ServicePointManager src/` invariant in CLAUDE.md is over-strict.** It matches doc comments that *reference* the forbidden API as anti-pattern guidance, not just usage. Future grep-based gates should match usage shape (`ServicePointManager\.\w+\s*=`) rather than the bare type name. Captured in SUMMARY-1.1 Lesson Seeds.
- **`global using` divergence between sibling test projects.** `FrigateRelay.Plugins.Doods2.Tests/Usings.cs` declares `global using FluentAssertions;` but `FrigateRelay.Plugins.CodeProjectAi.Tests/Usings.cs` does not. When mirroring a test file from a peer project, check the target project's `Usings.cs` rather than assuming the reference's convention carries over. The mirror-exactly intent is for test logic, not import boilerplate.
- **`StaticOptionsMonitor<T>` duplicated across 7 test files** (11 nested-class definitions) — same anti-pattern that produced the Phase 6 `CapturingLogger<T>` extraction. Recorded as the first Phase 17 cleanup task; CLAUDE.md updated to warn against new copies.
- **Tests written but never run leave bugs hidden.** The PLAN-1.2 builder reported "build succeeded, ready to commit" after the polling-site refactor, but the test suite caught the missed `validators.rejected` terminal state. Lesson: pre-commit `dotnet run` test sweep is a hard gate, not advisory — trust no agent's "looks good" without the test output.

### Process Improvements

- **Pre-commit verification sweep saved a regression and should be a hard gate** in the orchestrator's commit flow, not just a recommendation. The PLAN-1.2 `validators.rejected` bug would have shipped if the orchestrator hadn't run `dotnet run --project tests/FrigateRelay.Host.Tests -c Release` before staging the commit.
- **Builder Write-tool sandboxing for `.shipyard/phases/N/results/` files** appeared multiple times this phase. The orchestrator now writes SUMMARY/REVIEW/AUDIT/SIMPLIFICATION/DOCUMENTATION files inline based on the agent's final-message dump when the agent reports "Write tool blocked." Pattern noted for Phase 17 process improvements.
- **Sequential dispatch is the safe default when shared test files conflict between plans, even if all-disjoint front-matter says otherwise.** The verifier's F-1 WARN flagged the shared `CounterIncrementTests.cs` + `EventPumpSpanTests.cs` between PLAN-1.1 and PLAN-1.2; following the WARN's recommendation prevented merge conflicts entirely. Future architects: when verifier WARN flags a file overlap, treat it as required sequential dispatch ordering, not advisory.
- **Builder tool-budget caps need orchestrator backup, not just SendMessage resumption.** This phase had ~5 SendMessage resumptions across 3 plans + 4 quality-gate agents. When an agent hits the budget cap on its final write step (e.g., the SIMPLIFICATION dump), the cheapest path is for the orchestrator to take the agent's content from its final message and call `Write` directly, rather than pay another tool-budget cycle on a resumed agent.
- **Sandbox `dotnet` blocks for builder agents but not the orchestrator** — every builder agent this phase reported `dotnet build` and `dotnet run` were blocked. The orchestrator ran them and fed the output back via SendMessage. The pattern works but adds 1-2 round-trips per plan; documenting it in CLAUDE.md as expected agent behavior would set realistic expectations for Phase 17+.
